# Идеограф — Навигатор по идеям книг

Chrome-расширение для извлечения, структурирования и навигации по идеям из учебников и научных книг. Позволяет разбивать книгу на сеть связанных идей, отслеживать уровень освоения и быстро ориентироваться в материале.

## Проблема

Невозможно глубоко изучить все книги, которые хочется. Линейное чтение страница за страницей — медленно и неэффективно. Книги по одной теме (например, матанализу) на 80-90% пересекаются по содержанию. Ключевой навык — не прочитывать всё подряд, а **оперировать идеями**: понимать что уже знаешь, что новое, как идеи связаны между собой.

## Решение

Расширение, которое:
- Загружает PDF/DJVU книгу (через File System Access API, без хранения в IndexedDB)
- Анализирует выбранный диапазон страниц через LLM (текстовый или vision-режим)
- Извлекает идеи в структурированном виде (название, суть, тип, глубина, важность)
- Строит граф связей между идеями
- Позволяет отмечать статус каждой идеи (знакомо / не понятно / освоено)
- даёт быстрый обзор книги за 10-15 секунд по предисловию или оглавлению

## Ключевые принципы

1. **Идеи, не страницы** — книга рассматривается как набор идей со связями, а не линейный текст
2. **VLM-экстракция** — книги без текстового слоя (сканы, DJVU) обрабатываются через vision-модели
3. **Нет встроенного ридера** — чтение во внешнем приложении (zathura и др.), расширение — «мозговой центр»
4. **Результат на русском** — извлечение идей из книги на любом языке, результат всегда на русском
5. **Селективный анализ** — обрабатываются только выбранные пользователем участки, не вся книга целиком

## Установка и настройка

### Установка расширения в Chrome

Идеограф — это Chrome-расширение Manifest V3, которое заменяет стандартную страницу новой вкладки. Для установки потребуется собрать проект из исходного кода.

**Подготовка окружения:**

Убедитесь, что у вас установлены Node.js (версии 18+) и npm:

```bash
node --version   # должна быть >= 18
npm --version
```

**Сборка расширения:**

```bash
# Клонировать репозиторий
git clone https://github.com/NikasAl/ideograph.git
cd ideograph

# Установить зависимости
npm install

# Собрать расширение (dev-режим с sourcemaps)
npm run build

# Или запустить dev-сервер с hot-reload
npm run dev
```

После сборки в каталоге `dist/` появится собранное расширение.

**Установка в Chrome:**

1. Откройте Chrome и перейдите по адресу `chrome://extensions/`
2. В правом верхнем углу включите переключатель **«Режим разработчика»** (Developer mode)
3. Нажмите кнопку **«Загрузить распакованное расширение»** (Load unpacked)
4. Выберите каталог `dist/` проекта (или `dist/chrome-mv3-dev/` при dev-сборке)
5. Расширение появится в списке установленных расширений
6. **Откройте новую вкладку** — вы увидите интерфейс Идеографа

> **Важно**: после каждого изменения исходного кода нужно пересобрать проект (`npm run build`) и нажать кнопку обновления на карточке расширения в `chrome://extensions/`. В dev-режиме (`npm run dev`) обновление происходит автоматически.

**Проверка установки:**

- При открытии новой вкладки должен отображаться интерфейс Идеографа с заголовком «💡 Идеограф»
- В правом верхнем углу должна быть кнопка ⚙️ (настройки)
- Если вы видите стандартную страницу Chrome — убедитесь, что расширение включено и нет конфликтов с другими расширениями, перехватывающими новую вкладку

### Настройка API ключей

Идеограф использует LLM для извлечения идей из книг. Для работы необходим API-ключ хотя бы одного провайдера.

**Получение ключей:**

| Провайдер | Где получить ключ | Формат ключа |
|-----------|-------------------|-------------|
| **OpenRouter** | [openrouter.ai/keys](https://openrouter.ai/keys) → «Create Key» | `sk-or-v1-...` |
| **z-ai** | Предоставляется при регистрации на платформе | Зависит от формата провайдера |

**Ввод ключа:**

1. Откройте Идеограф (новая вкладка в Chrome)
2. Нажмите ⚙️ **Настройки** в правом верхнем углу
3. В разделе **«AI провайдеры»** введите API-ключ в соответствующее поле
4. В разделе **«Провайдер»** выберите активного провайдера из выпадающего списка
5. Нажмите **💾 Сохранить**

> **Безопасность**: API-ключи хранятся локально в IndexedDB браузера и никогда не отправляются на сторонние серверы (кроме самого API провайдера при выполнении запросов).

### Выбор моделей

Идеограф использует три модели для разных этапов обработки. Все модели настраиваются в разделе **«🧠 Модели для анализа»** в настройках:

| Параметр | Назначение | Рекомендация (OpenRouter) |
|----------|-----------|---------------------------|
| **Основная модель (activeModel)** | Извлечение идей из текста / Markdown. Должна хорошо понимать русский язык и работать с JSON. | `anthropic/claude-sonnet-4` — лучший баланс качества и цены |
| **OCR модель (ocrModel)** | Конвертация изображения страницы в Markdown с LaTeX-формулами. Vision-модель, нужна точность OCR, но не обязательно самая умная. | `google/gemini-2.0-flash-001` — быстрая, хорошая для OCR |
| **VLM модель (vlmModel)** | Полный визуальный анализ страницы — извлечение идей напрямую из изображения (геометрия, чертежи). Нужна сильная vision-модель. | `anthropic/claude-sonnet-4` или `google/gemini-2.5-pro-preview` |
| **Fallback модели (fallbackModels)** | Резервные модели, которые используются при ошибках основной модели (429 rate-limit, 502/503/529 server error). Через запятую. | `google/gemma-4-26b-a4b-it:free,openai/gpt-4o-mini,meta-llama/llama-3.1-70b-instruct` |

В настройках также есть кнопки-пресеты, которые автоматически заполняют поле соответствующей моделью.

**Почему три модели, а не одна:**

Разные этапы обработки требуют разных возможностей. OCR-конвертация изображения в текст — относительно простая задача, для которой достаточно быстрой и недорогой vision-модели (Gemini Flash). А вот извлечение идей из текста — задача, требующая глубокого понимания контекста и умения работать на русском языке, здесь лучше работает Claude. Для визуального анализа (чертежи, графики) нужна мощная vision-модель. Использование одной модели для всего было бы либо слишком дорогим (использовать Claude для OCR), либо недостаточно качественным (использовать Flash для извлечения идей).

**Fallback-модели:**

При частых ошибках (rate-limit 429, server errors) основная модель может быть недоступна. Список fallback-моделей позволяет автоматически переключаться на запасные варианты. Каждая fallback-модель получает до 2 попыток с экспоненциальной задержкой перед переходом к следующей.

### Подключение книг

1. В Идеографе нажмите **«+ Добавить книгу»**
2. В появившемся диалоге выберите PDF или DJVU файл
3. Расширение автоматически:
   - Определит формат файла
   - Посчитает количество страниц
   - Извлечёт текст с нескольких страниц (до 5 равномерно распределённых) и оценит качество текстового слоя
   - Подберёт оптимальный режим экстракции (текстовый / OCR / VLM)
4. Книга появится в библиотеке с указанием формата и рекомендованного режима

> **Ограничение**: файлы книг **не копируются** в базу данных расширения. Идеограф хранит только ссылку на файл (через File System Access API). Сам файл остаётся на диске. Если вы удалите или переместите файл, Идеограф не сможет получить к нему доступ — нужно будет переподключить файл через кнопку на карточке книги.

### Интеграция с zathura (Native Messaging Host)

Идеограф может автоматически открывать PDF в zathura на нужной странице при нажатии кнопки **«📖 zathura»** на карточке идеи. Для этого используется механизм Chrome [Native Messaging Host (NMH)](https://developer.chrome.com/docs/extensions/develop/concepts/native-messaging) — расширение отправляет JSON-сообщение локальному Python-скрипту, который запускает zathura.

**Архитектура:**

```
┌─────────────────────┐      JSON message       ┌──────────────────────┐
│   Chrome Extension   │  ───────────────────►  │  Native Host (Python) │
│   (idea-list.ts)     │                        │  ideograph-host.py    │
│                     │  ◄───────────────────  │                      │
│  sendNativeMessage() │      JSON response     │  subprocess.Popen()  │
└─────────────────────┘                        └──────────┬───────────┘
                                                          │
                                                          ▼
                                               ┌──────────────────────┐
                                               │    zathura -P 17     │
                                               │    --fork file.pdf   │
                                               └──────────────────────┘
```

**Установка NMH:**

```bash
cd native-host
bash install.sh
```

Скрипт `install.sh`:
- Копирует `ideograph-host.py` в `~/.local/share/ideograph/`
- Создаёт манифест `com.ideograph.host.json` в `~/.config/google-chrome/NativeMessagingHosts/` (и `chromium/`)
- Проверяет наличие Python3 и zathura

> **Важно:** после установки NMH перезапустите Chrome.

**Проверка работоспособности:**

```bash
python3 ~/.local/share/ideograph/ideograph-host.py
# Вставьте: {"action":"ping"} и нажмите Enter, Ctrl+D
# Должно вывести: {"status": "ok", "version": "1.0"}
```

**Zathura CLI опции, используемые Идеографом:**

| Опция | Описание | Пример |
|-------|----------|--------|
| `-P N` | Открыть на странице N (1-based) | `zathura -P 17 file.pdf` |
| `--fork` | Запустить в фоне (не блокировать терминал) | `zathura --fork file.pdf` |

> **Поиск в zathura:** стандартный zathura не имеет CLI-опции для поиска по тексту. После открытия на нужной странице нажмите `/` в zathura и введите фразу для поиска. Идеограф показывает подсказку с фразой из цитаты идеи.

**Fallback без NMH:**

Если Native Messaging Host не установлен, кнопка zathura копирует команду в буфер обмена. Пользователь вставляет её в терминал вручную. Команда выглядит так: `zathura -P 17 --fork '/path/to/file.pdf'`.

**Удаление NMH:**

```bash
rm -rf ~/.local/share/ideograph
rm ~/.config/google-chrome/NativeMessagingHosts/com.ideograph.host.json
rm ~/.config/chromium/NativeMessagingHosts/com.ideograph.host.json
```

---

> **Персистентность handle**: `FileSystemFileHandle` является structured-cloneable (Chrome 86+) и сохраняется в IndexedDB в отдельной таблице `fileHandles` (схема v2). После перезагрузки расширения или закрытия браузера handle автоматически восстанавливается из IndexedDB. Для повторного получения доступа используется `queryPermission()` / `requestPermission()`. Если permission не подтверждён — показывается кнопка «Переподключить».

### Модули

#### 1. Библиотека книг
- Список загруженных книг с metadata (название, автор, кол-во страниц, формат)
- Подключение книги через File System Access API (`showOpenFilePicker`)
- Персистентное хранение `FileSystemFileHandle` в IndexedDB (таблица `fileHandles`, ~100 байт на запись)
- После перезагрузки расширения — автоматическое восстановление handle из IndexedDB и запрос permission через `queryPermission()` / `requestPermission()`
- Автоопределение формата (PDF / DJVU)
- Если permission не подтверждён — кнопка «Переподключить»
- Интеграция с внешним ридером: запуск zathura через Native Messaging Host (NMH) с автоматическим открытием на нужной странице; fallback — копирование команды в буфер обмена

#### 2. Экстракция идей (3 режима)
- Выбор диапазона страниц для анализа
- Мультистраничная оценка качества текстового слоя (до 5 страниц, равномерно распределённых)
- Подробный отчёт по качеству каждой страницы
- Три режима экстракции (подробнее см. [Режимы экстракции](#режимы-экстракции)):
  - **Текстовый (text)**: PDF.js → текст → LLM. Быстро и дёшево.
  - **OCR + Текстовый (ocr)**: страница → изображение → vision LLM → Markdown с LaTeX → text LLM → идеи. Два вызова, формулы распознаются корректно.
  - **Полный визуальный (vlm)**: страница → изображение → vision LLM → идеи напрямую. Для геометрии, чертежей, графиков.
- Автоопределение режима по качеству текстового слоя (0-100%)
- Пользователь может переключить режим вручную
- **Выбранный пользователем режим всегда учитывается** — пайплайн не переопределяет режим на основе повторной оценки

#### 3. Карточки идей
- Список идей с фильтрами (по статусу, глубине, типу, книге)
- Карточка каждой идеи: название, суть, тип, важность, страница
- Интерактивные статусы: Не изучал → В процессе → Освоено → Применяю
- Знакомство: Знаю глубоко / Слышал / Не знаком
- Пользовательские заметки и вопросы к каждой идее

## Режимы экстракции

Идеограф поддерживает три режима извлечения идей из страниц книги. Режим выбирается автоматически на основе оценки качества текстового слоя, но пользователь может переключить его вручную.

### Текстовый режим (text)

**Когда использовать:** текстовый слой PDF хорошего качества — текст читается, формулы корректны (или отсутствуют), OCR-ошибок нет.

**Как работает:**
1. PDF.js извлекает текст с каждой страницы (textContent)
2. Текст разбивается на чанки (~4000 символов, с учётом границ страниц)
3. Каждый чанк отправляется в **текстовую LLM** для извлечения идей
4. Извлечённые идеи дедуплицируются
5. Все идеи отправляются в LLM для построения связей (проход 2)

**Преимущества:** быстро, дёшево (1 вызов LLM на чанк), хорошо работает для большинства современных PDF с текстовым слоем.

**Ограничения:** не подходит для книг с повреждённым текстовым слоем, сканов, DJVU без текста, книг с формулами в виде картинок.

### OCR + Текстовый анализ (ocr)

**Когда использовать:** текстовый слой есть, но некачественный — формулы превратились в набор символов, OCR-ошибки в тексте, частично повреждённый скан. Типично для старых сканированных PDF и DJVU с текстовым слоем.

**Как работает:**
1. **Фаза 1 — OCR-конвертация:**
   - Каждая страница рендерится в изображение (OffscreenCanvas → PNG)
   - Изображение отправляется в **vision LLM** (OCR модель) с инструкцией конвертировать в Markdown
   - Vision LLM восстанавливает формулы в LaTeX-нотации: `$$...$$` для блочных, `$...$` для inline
   - Результат кэшируется в `PageTextCache.ocrMarkdown`
2. **Фаза 2 — извлечение идей:**
   - Полученный Markdown разбивается на чанки
   - Каждый чанк отправляется в **текстовую LLM** для извлечения идей
   - Дедупликация + построение связей (как в текстовом режиме)

**Преимущества:** формулы распознаются корректно в LaTeX, текстовая LLM работает дешевле vision для анализа идей, результат — структурированный Markdown, пригодный для повторного использования.

**Ограничения:** два вызова LLM на страницу (OCR + анализ), дороже текстового режима, зависит от качества vision модели.

### Полный визуальный анализ (vlm)

**Когда использовать:** принципиально визуальный контент — книги по геометрии с чертежами, графами, диаграммами. Также для книг без текстового слоя (чистые сканы, DJVU).

**Как работает:**
1. Каждая страница рендерится в изображение (OffscreenCanvas → PNG)
2. Изображение + промпт отправляются в **vision LLM** (VLM модель)
3. Vision LLM анализирует и текст, и изображения (чертежи, графики, формулы) и извлекает идеи
4. Дедупликация + построение связей

**Преимущества:** один вызов на страницу, понимает визуальный контент (чертежи, графики), не зависит от качества текстового слоя.

**Ограничения:** дороже (vision модели стоят больше), требует мощную vision модель (Claude, GPT-4o, Gemini Pro), может хуже работать с длинными текстовыми страницами.

### Предпросмотр текстового слоя

При нажатии «Анализировать» Идеограф оценивает качество текстового слоя **по нескольким страницам** (до 5 страниц, равномерно распределённых по выбранному диапазону), а не только по первой. Это позволяет более точно определить режим экстракции — в некоторых книгах первые страницы (оглавление, предисловие) могут иметь хороший текст, а основной текст — быть сканом.

```
┌─────────────────────────────────────────────────────────────┐
│  Оценка текстового слоя (5 страниц):                         │
│                                                             │
│  стр. 3   ████████████████████░░  85%  ✓ хорошо            │
│  стр. 25  ████████████████████░░  82%  ✓ хорошо            │
│  стр. 48  ██░░░░░░░░░░░░░░░░░░░  15%  ✗ плохо (скан)      │
│  стр. 71  █░░░░░░░░░░░░░░░░░░░░   8%  ✗ плохо (скан)      │
│  стр. 94  ░░░░░░░░░░░░░░░░░░░░░   3%  ✗ нет текста        │
│                                                             │
│  Среднее: 39%  →  Рекомендация: OCR + Текстовый анализ      │
│                                                             │
│  ┌───────────────────────────────────────────────────────┐  │
│  │ ГЛАВА 3. ПРЕДЕЛЫ ФУНКЦИЙ                              │  │
│  │                                                        │  │
│  │ Определение 3.1. Пусть функция f(x)                   │  │
│  │ определена на множестве E...                           │  │
│  └───────────────────────────────────────────────────────┘  │
│                                                             │
│  Режим: [Текстовый] [OCR + Текстовый ▼] [VLM]               │
└─────────────────────────────────────────────────────────────┘
```

На основе этого предпросмотра пользователь принимает решение:
- **Текст хороший** → режим «Текстовый» (быстро, дёшево)
- **Текст плохой, формулы нечитаемые** → режим «OCR + Текстовый» (формулы в LaTeX)
- **Чертежи, визуальный контент** → режим «VLM» (vision анализ)

Алгоритм оценки качества (`mode-detector.ts`) анализирует:
- Долю читаемых символов (буквы, цифры)
- Наличие мусорных/нечитаемых символов
- Долю пробелов (слишком много — пустая страница)
- Обрывки слов (одиночные буквы — признак плохого OCR)
- Математические символы вне формул (признак повреждённых формул)
- Общий объём текста

> **Важно:** пользовательский выбор режима **всегда учитывается** пайплайном. Пайплайн больше не переопределяет режим на основе повторной оценки отдельной страницы. UI уже провёл мультистраничную оценку и дал рекомендацию — решение за пользователем.

## Работа с оглавлением (TOC)

### Концепция: оглавление как каркас идей

Оглавление — это не просто список разделов для навигации. В контексте Идеографа оглавление играет роль **каркаса идей книги**: оно задаёт иерархическую структуру, привязанную к диапазонам страниц, к которому затем подключаются извлечённые идеи. Это позволяет пользователю видеть книгу не как линейный набор страниц, а как дерево тематических блоков, каждый со своими идеями, прогрессом изучения и связями.

В отличие от простого извлечения TOC из PDF-метаданных (которое есть далеко не во всех книгах, особенно в сканированных учебниках), Идеограф использует LLM для распознавания оглавления с текстового или визуального слоя книги. Пользователь вручную указывает, на каких страницах находится оглавление — это даёт максимальную гибкость и работает для любых книг: от современных PDF с текстовым слоем до старых сканированных DJVU.

**Ключевые отличия от стандартного подхода:**

1. **Ручное указание страниц оглавления** — пользователь знает, где в конкретной книге напечатано оглавление. Не нужно автообнаружение, которое часто ошибается.
2. **LLM-распознавание** — оглавление может быть оформлено по-разному (римские цифры, точки, многоколоночная вёрстка). LLM справляется с разнообразными форматами.
3. **Скелет идей** — каждой главе назначается краткое описание содержания. Это позволяет получить обзор книги за 15 секунд, не читая её целиком.
4. **Привязка идей к главам** — извлечённые идеи автоматически связываются с соответствующими главами через `chapterId`.
5. **Batch-анализ по главам** — после извлечения оглавления пользователь может анализировать книгу глава за главой, одной кнопкой.

```
    - Извлечённые идеи привязываются к главам (chapterId)
    - Фильтрация идей по главам
    - На графе — кластеры по главам
```

### Извлечение оглавления через LLM

**Режим извлечения TOC определяется режимом книги:**

| Режим книги | Источник данных для TOC | Модель |
|------------|------------------------|--------|
| `text` | Текстовый слой PDF.js (textContent со страниц оглавления) | `activeModel` (текстовая) |
| `ocr` | Рендер страниц → Vision LLM → Markdown → Текстовая LLM | `ocrModel` → `activeModel` |
| `vlm` | Рендер страниц → Vision LLM (напрямую) | `vlmModel` |

**Многостраничное оглавление:** Если оглавление занимает несколько страниц, все они объединяются в один запрос (или в несколько запросов с последующим мержем). LLM получает контекст всех страниц и возвращает единое оглавление.

**PDF-метаданные (дополнение):** Некоторые PDF содержат встроенное оглавление (PDF outlines). Идеограф может извлечь его через `pdf.getOutline()` (PDF.js) и показать пользователю как вариант для сравнения или как дополнение к LLM-результату. Но основной механизм — именно LLM-распознавание, потому что:
- PDF outlines часто отсутствуют в сканированных книгах
- PDF outlines могут быть неполными или некорректными
- LLM видит фактическое визуальное оглавление с точными номерами страниц

**Валидация результата:** После извлечения Идеограф проверяет:
- Все `page` значения валидны (в пределах 1..totalPages)
- Нет дубликатов заголовков на одном уровне
- `parentId` ссылается на существующий entry
- Иерархия корректна (нет циклов, level увеличивается не более чем на 1)

### Вычисление диапазонов страниц

Для каждого `TOCEntry` автоматически вычисляется `pageEnd`:

```typescript
function computePageRanges(entries: TOCEntry[], totalPages: number): void {
  // Сортировка по странице, затем по уровню (глубокие сначала)
  const sorted = [...entries].sort((a, b) => {
    if (a.page !== b.page) return a.page - b.page;
    return a.level - b.level;
  });

  for (let i = 0; i < sorted.length; i++) {
    const entry = sorted[i];
    // Ищем следующий entry на том же или более высоком уровне
    let nextEntry: TOCEntry | undefined;
    for (let j = i + 1; j < sorted.length; j++) {
      if (sorted[j].level <= entry.level) {
        nextEntry = sorted[j];
        break;
      }
    }
    entry.pageEnd = nextEntry ? nextEntry.page - 1 : totalPages;
  }
}
```

**Пример:**

```
Глава 1. Пределы (стр. 6, level 1)       → pageEnd = 17 (перед главой 2)
  1.1 Определение (стр. 6, level 2)       → pageEnd = 11 (перед 1.2)
  1.2 Свойства (стр. 12, level 2)         → pageEnd = 17 (перед главой 2)
Глава 2. Непрерывность (стр. 18, level 1) → pageEnd = 29 (перед главой 3)
...
Глава N (последняя)                        → pageEnd = totalPages
```

Для entry верхнего уровня (level 1, глава) `pageEnd` — это страница перед следующей главой. Для подразделов (level 2, 3) — перед следующим подразделом того же уровня. Последний entry получает `pageEnd = totalPages`.

### Привязка идей к главам

При экстракции идей пайплайн автоматически определяет, к какой главе принадлежит каждая страница, и устанавливает `idea.chapterId`:

```typescript
function findChapterForPage(page: number, toc: TOCEntry[]): TOCEntry | undefined {
  // Ищем entry level 1 (глава), чей диапазон содержит страницу
  return toc
    .filter(e => e.level === 1 && e.pageEnd !== undefined)
    .find(e => page >= e.page && page <= e.pageEnd!);
}
```

**Принцип:** каждая идея привязывается к главе верхнего уровня (level 1). Подразделы (level 2, 3) используются для детализации в UI и в промпте-контексте, но `chapterId` всегда указывает на главу. Это упрощает фильтрацию и статистику.

**Контекст главы в промптах:** При извлечении идей из диапазона страниц пайплайн добавляет в промпт заголовок текущей главы и краткое описание (summary). Это помогает LLM лучше понимать контекст и извлекать более релевантные идеи.

**Обновление ideasCount:** После каждого анализа пайплайн пересчитывает `ideasCount` для всех затронутых глав TOC.

## Обработка ошибок и fallback

AI-клиент реализует механизм fallback с повторными попытками для обеспечения надёжности анализа.

### Классификация ошибок

| Тип | Коды / признаки | Поведение |
|-----|----------------|-----------|
| **Временные (transient)** | HTTP 429 (rate-limit), 502, 503, 529 | Повторная попытка с той же моделью, exponential backoff (до 2 повторов, задержка 1s → 2s → 4s, макс 10s) |
| **Постоянные для модели** | HTTP 400, 401, 403, 422; «Provider returned error» | Немедленный переход к следующей fallback-модели без повторов |
| **Исчерпаны попытки** | Модель повторила ошибку 3 раза подряд | Переход к следующей fallback-модели |

### Порядок выбора моделей

1. **Основная модель** — модель, соответствующая текущему режиму (`activeModel`, `ocrModel` или `vlmModel`)
2. **Fallback-модели** — модели из настройки `fallbackModels` (запятые-разделённый список)
3. Если все модели исчерпаны — ошибка записывается в `analysisLog` с `status: 'error'`

### Сохранение partial results

При ошибке пайплайн сохраняет все уже извлечённые идеи в IndexedDB и обновляет `book.lastAnalyzedPage`. Пользователь может запустить анализ повторно — он продолжится с той страницы, на которой остановился.

### Analysis log

Каждый запуск анализа записывается в таблицу `analysisLog`:

```typescript
interface AnalysisLog {
  id?: number;
  bookId: string;
  pageFrom: number;
  pageTo: number;
  mode: 'text' | 'ocr' | 'vlm';
  provider: string;
  model: string;
  ideasCount: number;
  relationsCount: number;
  tokensUsed?: number;
  startedAt: number;
  completedAt: number;
  status: 'success' | 'error' | 'cancelled';
  error?: string;                // описание ошибки при status = 'error'
}
```

Это позволяет отслеживать историю анализа и диагностировать проблемы.

## Спецификация LLM-промптов

> Подробные промпты будут в `docs/prompts.md` после фазы 0. Ниже — общая стратегия.

### Проход 1: Извлечение идей (текстовый режим)

```
Системный промпт:
  Ты — аналитик знаний. Извлекай из текста самостоятельные идеи,
  концепции, определения, утверждения, методы.
  Пиши результат на русском языке.

  Для каждой идеи укажи:
  - title: краткое название (до 10 слов)
  - summary: суть в 1-2 предложениях
  - type: definition | method | theorem | insight | example | analogy
  - depth: basic | medium | advanced
  - importance: 1-5 (насколько фундаментальна)
  - pages: номера страниц (из контекста)
  - quote: ключевая цитата (если есть)

  Правила:
  - НЕ дроби мелкие детали
  - НЕ дублируй повторяющиеся идеи
  - Если идея опирается на другую — укажи в requires[]

  Верни JSON массив идей.
```

### Проход 1: Извлечение идей (VLM-режим)

```
Системный промпт:
  Ты — аналитик знаний. На изображении — страница учебника/книги.
  Извлеки все идеи с этой страницы.
  Пиши результат на русском языке.
  Формат тот же, что и для текстового режима.
```

### Проход 2: Построение связей

```
Системный промпт:
  Дан список идей из книги. Определи связи между ними:

  Типы связей:
  - prerequisite: идея A нужна для понимания B
  - elaborates: идея A развивает/уточняет B
  - contradicts: идея A противоречит B
  - analogous: идея A аналогична B из другой области
  - applies: идея A применяется в B

  Верни JSON массив: [{source, target, type, description}]
```

## Решения по ключевым вопросам

### Хранение файлов: File System Access API

**Почему не IndexedDB для файлов:**
- Лимит IndexedDB: 50-500MB (зависит от браузера и диска)
- Типичный учебник: 10-100MB, DJVU: 5-50MB
- 10-20 книг могут заполнить лимит

**Почему не Native Messaging:**
- Требует установки отдельной программы на компьютер
- Разная установка под Windows/Linux/macOS
- Избыточно для нашей задачи

**Решение:** `FileSystemFileHandle` хранится в IndexedDB (~100 байт) в отдельной таблице `fileHandles` (схема v2, миграция). `FileSystemFileHandle` является structured-cloneable (Chrome 86+), поэтому корректно сохраняется и восстанавливается из IndexedDB. При необходимости чтения — `handle.getFile()` → ArrayBuffer → рендер. Книга остаётся на диске, IndexedDB не перегружается.

**Восстановление после перезагрузки:** при загрузке расширения handle автоматически извлекается из IndexedDB (файл `src/ui/utils/file-store.ts`). Для получения доступа используется `queryPermission()`: если permission уже подтверждён — доступ восстанавливается мгновенно, без всплывающего окна. Если permission не подтверждён — вызывается `requestPermission()` и показывается системный диалог. Если пользователь отказал — показывается кнопка «Переподключить».

**filePath для zathura:** поле `Book.filePath` сохраняется при подключении книги и используется для генерации команды `zathura -f <page> <filepath>`. Путь хранится для справки, но через `resolve()` нельзя восстановить handle по пути — поэтому handle хранится отдельно в `fileHandles`.

### DJVU-рендер: стратегия

1. **Основной путь**: djvu.js (JavaScript-библиотека для рендера в canvas)
2. **Fallback**: Если djvu.js не справляется с конкретным файлом — предложить пользователю конвертацию через Linux-утилиты:
   - `djvups book.djvu book.ps && ps2pdf book.ps book.pdf`
   - `ddjvu -format=pdf book.djvu book.pdf`
3. **OCR fallback**: Если ни текстовый слой, ни VLM не дают результат — предложить `tesseract` через Native Messaging (далёкая перспектива)

### Интеграция с внешним ридером (zathura)

Каждая идея содержит кнопку **«📖 Открыть в zathura»** → копирует в буфер обмена команду:

```bash
zathura -f 42 /home/user/books/math_analysis.djvu
```

Пользователь вставляет в терминал одним кликом. Не требует установки native host, работает на любом Linux.

Если идея содержит `quote` (ключевую цитату), в тултипе кнопки отображается подсказка для поиска по тексту в zathura.

В перспективе: можно зарегистрировать `ideograph://` protocol handler через `registerProtocolHandler()`, но это требует desktop-файла и более сложной настройки.

### Мультиязычность книг

Язык книги не важен. В промпте указано: «пиши результат на русском языке». LLM (особенно GPT-4o, Claude) отлично извлекает идеи с английского, немецкого, французского и других языков и формулирует на русском. Не нужно делать перевод книги — только извлечение идей на русском.

### Автоопределение режима (text / OCR / VLM)

Алгоритм оценки качества текстового слоя (`mode-detector.ts`) анализирует извлечённый текст с **нескольких страниц** (до 5, равномерно распределённых по выбранному диапазону) и возвращает числовую оценку (0-1) для каждой страницы, среднюю оценку и рекомендацию режима. Упрощённая логика:

```
Для каждой страницы:
  score = доля читаемых символов

  Если garbage-символов > 10%   → score -= garbage * 0.5
  Если обрывков слов > 30%      → score -= 0.3
  Если пробелов > 85%          → score -= 0.3
  Если математ. символов > 3   → score -= 0.15

Средний score по всем страницам:
  score < 0.3  → VLM (текстовый слой непригоден)
  score < 0.65 → OCR (текст частично повреждён, формулы)
  иначе         → TEXT (текст хорошего качества)

DJVU-штраф: если формат djvu и score < 0.8 → OCR
```

**Важно:** пользователь всегда может изменить рекомендованный режим вручную. После того как пользователь выбрал режим, пайплайн **не переопределяет** его. Выбранный режим используется «как есть» на протяжении всего анализа.
