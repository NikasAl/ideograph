# Идеограф — Навигатор по идеям книг

Chrome-расширение для извлечения, структурирования и навигации по идеям из учебников и научных книг. Позволяет разбивать книгу на сеть связанных идей, отслеживать уровень освоения и быстро ориентироваться в материале.

## Проблема

Невозможно глубоко изучить все книги, которые хочется. Линейное чтение страница за страницей — медленно и неэффективно. Книги по одной теме (например, матанализу) на 80-90% пересекаются по содержанию. Ключевой навык — не прочитывать всё подряд, а **оперировать идеями**: понимать что уже знаешь, что новое, как идеи связаны между собой.

## Решение

Расширение, которое:
- Загружает PDF/DJVU книгу (через File System Access API, без хранения в IndexedDB)
- Анализирует выбранный диапазон страниц через LLM (текстовый или vision-режим)
- Извлекает идеи в структурированном виде (название, суть, тип, глубина, важность)
- Строит граф связей между идеями
- Позволяет отмечать статус каждой идеи (знакомо / не понятно / освоено)
- даёт быстрый обзор книги за 10-15 секунд по предисловию или оглавлению

## Ключевые принципы

1. **Идеи, не страницы** — книга рассматривается как набор идей со связями, а не линейный текст
2. **VLM-экстракция** — книги без текстового слоя (сканы, DJVU) обрабатываются через vision-модели
3. **Нет встроенного ридера** — чтение во внешнем приложении (zathura и др.), расширение — «мозговой центр»
4. **Результат на русском** — извлечение идей из книги на любом языке, результат всегда на русском
5. **Селективный анализ** — обрабатываются только выбранные пользователем участки, не вся книга целиком

## Установка и настройка

### Установка расширения в Chrome

Идеограф — это Chrome-расширение Manifest V3, которое заменяет стандартную страницу новой вкладки. Для установки потребуется собрать проект из исходного кода.

**Подготовка окружения:**

Убедитесь, что у вас установлены Node.js (версии 18+) и npm:

```bash
node --version   # должна быть >= 18
npm --version
```

**Сборка расширения:**

```bash
# Клонировать репозиторий
git clone https://github.com/NikasAl/ideograph.git
cd ideograph

# Установить зависимости
npm install

# Собрать расширение (dev-режим с sourcemaps)
npm run build

# Или запустить dev-сервер с hot-reload
npm run dev
```

После сборки в каталоге `dist/` появится собранное расширение.

**Установка в Chrome:**

1. Откройте Chrome и перейдите по адресу `chrome://extensions/`
2. В правом верхнем углу включите переключатель **«Режим разработчика»** (Developer mode)
3. Нажмите кнопку **«Загрузить распакованное расширение»** (Load unpacked)
4. Выберите каталог `dist/` проекта (или `dist/chrome-mv3-dev/` при dev-сборке)
5. Расширение появится в списке установленных расширений
6. **Откройте новую вкладку** — вы увидите интерфейс Идеографа

> **Важно**: после каждого изменения исходного кода нужно пересобрать проект (`npm run build`) и нажать кнопку обновления на карточке расширения в `chrome://extensions/`. В dev-режиме (`npm run dev`) обновление происходит автоматически.

**Проверка установки:**

- При открытии новой вкладки должен отображаться интерфейс Идеографа с заголовком «💡 Идеограф»
- В правом верхнем углу должна быть кнопка ⚙️ (настройки)
- Если вы видите стандартную страницу Chrome — убедитесь, что расширение включено и нет конфликтов с другими расширениями, перехватывающими новую вкладку

### Настройка API ключей

Идеограф использует LLM для извлечения идей из книг. Для работы необходим API-ключ хотя бы одного провайдера.

**Получение ключей:**

| Провайдер | Где получить ключ | Формат ключа |
|-----------|-------------------|-------------|
| **OpenRouter** | [openrouter.ai/keys](https://openrouter.ai/keys) → «Create Key» | `sk-or-v1-...` |
| **z-ai** | Предоставляется при регистрации на платформе | Зависит от формата провайдера |

**Ввод ключа:**

1. Откройте Идеограф (новая вкладка в Chrome)
2. Нажмите ⚙️ **Настройки** в правом верхнем углу
3. В разделе **«AI провайдеры»** введите API-ключ в соответствующее поле
4. В разделе **«Провайдер»** выберите активного провайдера из выпадающего списка
5. Нажмите **💾 Сохранить**

> **Безопасность**: API-ключи хранятся локально в IndexedDB браузера и никогда не отправляются на сторонние серверы (кроме самого API провайдера при выполнении запросов).

### Выбор моделей

Идеограф использует три модели для разных этапов обработки. Все модели настраиваются в разделе **«🧠 Модели для анализа»** в настройках:

| Параметр | Назначение | Рекомендация (OpenRouter) |
|----------|-----------|---------------------------|
| **Основная модель** | Извлечение идей из текста / Markdown. Должна хорошо понимать русский язык и работать с JSON. | `anthropic/claude-sonnet-4` — лучший баланс качества и цены |
| **OCR модель** | Конвертация изображения страницы в Markdown с LaTeX-формулами. Vision-модель, нужна точность OCR, но не обязательно самая умная. | `google/gemini-2.0-flash-001` — быстрая, хорошая для OCR |
| **VLM модель** | Полный визуальный анализ страницы — извлечение идей напрямую из изображения (геометрия, чертежи). Нужна сильная vision-модель. | `anthropic/claude-sonnet-4` или `google/gemini-2.5-pro-preview` |

В настройках также есть кнопки-пресеты, которые автоматически заполняют поле соответствующей моделью.

**Почему три модели, а не одна:**

Разные этапы обработки требуют разных возможностей. OCR-конвертация изображения в текст — относительно простая задача, для которой достаточно быстрой и недорогой vision-модели (Gemini Flash). А вот извлечение идей из текста — задача, требующая глубокого понимания контекста и умения работать на русском языке, здесь лучше работает Claude. Для визуального анализа (чертежи, графики) нужна мощная vision-модель. Использование одной модели для всего было бы либо слишком дорогим (использовать Claude для OCR), либо недостаточно качественным (использовать Flash для извлечения идей).

### Подключение книг

1. В Идеографе нажмите **«+ Добавить книгу»**
2. В появившемся диалоге выберите PDF или DJVU файл
3. Расширение автоматически:
   - Определит формат файла
   - Посчитает количество страниц
   - Извлечёт текст с первой страницы и оценит его качество
   - Подберёт оптимальный режим экстракции (текстовый / OCR / VLM)
4. Книга появится в библиотеке с указанием формата и рекомендованного режима

> **Ограничение**: файлы книг **не копируются** в базу данных расширения. Идеограф хранит только ссылку на файл (через File System Access API). Сам файл остаётся на диске. Если вы удалите или переместите файл, Идеограф не сможет получить к нему доступ — нужно будет переподключить файл через кнопку на карточке книги.

> **Permission**: разрешение на чтение файла сохраняется пока открыт браузер. После перезапуска Chrome может потребоваться переподключить файл (кнопка «Переподключить» на карточке книги).

### Модули

#### 1. Библиотека книг
- Список загруженных книг с metadata (название, автор, кол-во страниц, формат)
- Подключение книги через File System Access API (`showOpenFilePicker`)
- Хранение `FileSystemFileHandle` в IndexedDB (весит ~100 байт, не файл)
- Автоопределение формата (PDF / DJVU)
- Если permission истёк — кнопка «Переподключить»
- Интеграция с внешним ридером: копирование команды `zathura -f N /path/to/file`

#### 2. Экстракция идей (3 режима)
- Выбор диапазона страниц для анализа
- Предпросмотр текстового слоя первой страницы с оценкой качества
- Три режима экстракции (подробнее см. [Режимы экстракции](#режимы-экстракции)):
  - **Текстовый (text)**: PDF.js → текст → LLM. Быстро и дёшево.
  - **OCR + Текстовый (ocr)**: страница → изображение → vision LLM → Markdown с LaTeX → text LLM → идеи. Два вызова, формулы распознаются корректно.
  - **Полный визуальный (vlm)**: страница → изображение → vision LLM → идеи напрямую. Для геометрии, чертежей, графиков.
- Автоопределение режима по качеству текстового слоя (0-100%)
- Пользователь может переключить режим вручную

#### 3. Карточки идей
- Список идей с фильтрами (по статусу, глубине, типу, книге)
- Карточка каждой идеи: название, суть, тип, важность, страница
- Интерактивные статусы: Не изучал → В процессе → Освоено → Применяю
- Знакомство: Знаю глубоко / Слышал / Не знаком
- Пользовательские заметки и вопросы к каждой идее
- Переход к странице книги: копирование команды для внешнего ридера

#### 4. Граф идей
- Force-directed граф на vis-network
- Визуальные кодировки:
  - **Форма узла**: ● определение, ■ метод, ◆ теорема, ▲ инсайт
  - **Цвет узла**: 🟢 освоено, 🟡 в процессе, 🔵 знакомо, ⚪ не изучал, 🔴 не понятно
  - **Размер**: важность × (3 - уровень знакомства)
  - **Толщина ребра**: 1px аналог, 2px развивает/применяет, 3px prerequisite
- Клик по узлу → карточка идеи
- Drag-and-drop, zoom/pan
- Фильтры и поиск

#### 5. Настройки (модальное окно)
- API ключи: OpenRouter, z-ai (расширяемо на GigaChat, Kimi, MiniMax)
- Выбор активного провайдера и модели
- Тема: светлая / тёмная / системная
- Язык интерфейса
- Детализация извлечения: низкая / средняя / высокая
- Экспорт/импорт базы идей

### Модель данных

```typescript
// ============ Книга ============
interface Book {
  id: string;                    // UUID
  title: string;                 // из metadata или имя файла
  author: string;                // из metadata

  format: 'pdf' | 'djvu';       // автоопределённый формат
  extractionMode: 'text' | 'ocr' | 'vlm'; // три режима (см. Режимы экстракции)

  fileHandle?: FileSystemFileHandle; // для File System Access API
  tableOfContents: TOCEntry[];   // оглавление (если доступно)

  updatedAt: number;

  page: number;
  level: number;                 // 1 = глава, 2 = секция, 3 = подсекция

  bookId: string;
  chapterId?: string;            // привязка к главе

  // Содержание
  title: string;                 // краткое название (до 10 слов)
  summary: string;               // суть в 1-2 предложениях
  quote?: string;                // ключевая цитата (если есть текстовый слой)

  // Классификация
  type: IdeaType;
  depth: IdeaDepth;
  importance: IdeaImportance;

  // Позиция в книге
  pages: number[];               // на каких страницах упоминается

  // Состояние пользователя
  familiarity: Familiarity;
  status: IdeaStatus;

  // Пользовательский контент
  notes: string;
  questions: string[];
  userTags: string[];

  // AI-мета
  aiModel: string;               // какая модель извлекла
  provider: string;              // OpenRouter, z-ai и т.д.
  extractedAt: number;

  // Связи (denormalized для быстрого доступа)
  relations: Relation[];
}

interface Relation {
  targetId: string;
  type: RelationType;
  description?: string;          // краткое описание связи
}

// ============ Настройки ============
interface Settings {
  activeProvider: 'openrouter' | 'z-ai';  // расширяемо
  providerKeys: {
    openrouter?: string;
    'z-ai'?: string;
    gigachat?: string;
    kimi?: string;
    minimax?: string;
  };
  activeModel: string;           // основная модель для извлечения идей (text)
  ocrModel: string;             // vision модель для OCR → Markdown+LaTeX
  vlmModel: string;             // vision модель для полного визуального анализа
  theme: 'light' | 'dark' | 'system';
  language: 'ru';
  extractionDetail: 'low' | 'medium' | 'high';
}

// ============ Кэш текста ============
interface PageTextCache {
  bookId: string;
  pageNumber: number;
  text: string;                  // извлечённый текст (если есть)
  hasTextLayer: boolean;         // есть ли текстовый слой
  imageBase64?: string;          // рендер страницы (для VLM-режима)
  cachedAt: number;
}
```

## Режимы экстракции

Идеограф поддерживает три режима извлечения идей из страниц книги. Режим выбирается автоматически на основе оценки качества текстового слоя, но пользователь может переключить его вручную.

### Текстовый режим (text)

**Когда использовать:** текстовый слой PDF хорошего качества — текст читается, формулы корректны (или отсутствуют), OCR-ошибок нет.

**Как работает:**
1. PDF.js извлекает текст с каждой страницы (textContent)
2. Текст разбивается на чанки (~4000 символов, с учётом границ страниц)
3. Каждый чанк отправляется в **текстовую LLM** для извлечения идей
4. Извлечённые идеи дедуплицируются
5. Все идеи отправляются в LLM для построения связей (проход 2)

**Преимущества:** быстро, дёшево (1 вызов LLM на чанк), хорошо работает для большинства современных PDF с текстовым слоем.

**Ограничения:** не подходит для книг с повреждённым текстовым слоем, сканов, DJVU без текста, книг с формулами в виде картинок.

### OCR + Текстовый анализ (ocr)

**Когда использовать:** текстовый слой есть, но некачественный — формулы превратились в набор символов, OCR-ошибки в тексте, частично повреждённый скан. Типично для старых сканированных PDF и DJVU с текстовым слоем.

**Как работает:**
1. **Фаза 1 — OCR-конвертация:**
   - Каждая страница рендерится в изображение (OffscreenCanvas → PNG)
   - Изображение отправляется в **vision LLM** (OCR модель) с инструкцией конвертировать в Markdown
   - Vision LLM восстанавливает формулы в LaTeX-нотации: `$$...$$` для блочных, `$...$` для inline
   - Результат кэшируется (Markdown с LaTeX)
2. **Фаза 2 — извлечение идей:**
   - Полученный Markdown разбивается на чанки
   - Каждый чанк отправляется в **текстовую LLM** для извлечения идей
   - Дедупликация + построение связей (как в текстовом режиме)

**Преимущества:** формулы распознаются корректно в LaTeX, текстовая LLM работает дешевле vision для анализа идей, результат — структурированный Markdown, пригодный для повторного использования.

**Ограничения:** два вызова LLM на страницу (OCR + анализ), дороже текстового режима, зависит от качества vision модели.

### Полный визуальный анализ (vlm)

**Когда использовать:** принципиально визуальный контент — книги по геометрии с чертежами, графами, диаграммами. Также для книг без текстового слоя (чистые сканы, DJVU).

**Как работает:**
1. Каждая страница рендерится в изображение (OffscreenCanvas → PNG)
2. Изображение + промпт отправляются в **vision LLM** (VLM модель)
3. Vision LLM анализирует и текст, и изображения (чертежи, графики, формулы) и извлекает идеи
4. Дедупликация + построение связей

**Преимущества:** один вызов на страницу, понимает визуальный контент (чертежи, графики), не зависит от качества текстового слоя.

**Ограничения:** дороже (vision модели стоят больше), требует мощную vision модель (Claude, GPT-4o, Gemini Pro), может хуже работать с длинными текстовыми страницами.

### Предпросмотр текстового слоя

При нажатии «Анализировать» Идеограф показывает предпросмотр текста первой страницы выбранного диапазона с оценкой качества:

```
┌─────────────────────────────────────────────────────┐
│  Предпросмотр текстового слоя (стр. 1):  Кач: 85%  │
│                                                     │
│  ┌───────────────────────────────────────────────┐  │
│  │ ГЛАВА 3. ПРЕДЕЛЫ ФУНКЦИЙ                      │  │
│  │                                                │  │
│  │ Определение 3.1. Пусть функция f(x)           │  │
│  │ определена на множестве E...                   │  │
│  └───────────────────────────────────────────────┘  │
│                                                     │
│  Рекомендация: Текстовый слой хорошего качества     │
└─────────────────────────────────────────────────────┘
```

На основе этого предпросмотра пользователь принимает решение:
- **Текст хороший** → режим «Текстовый» (быстро, дёшево)
- **Текст плохой, формулы нечитаемые** → режим «OCR + Текстовый» (формулы в LaTeX)
- **Чертежи, визуальный контент** → режим «VLM» (vision анализ)

Алгоритм оценки качества (`mode-detector.ts`) анализирует:
- Долю читаемых символов (буквы, цифры)
- Наличие мусорных/нечитаемых символов
- Долю пробелов (слишком много — пустая страница)
- Обрывки слов (одиночные буквы — признак плохого OCR)
- Математические символы вне формул (признак повреждённых формул)
- Общий объём текста

## Пайплайн обработки

```
 1. Пользователь подключает книгу (File System Access API)
      │
      ▼
 2. Определение формата (PDF/DJVU) и количества страниц
      │
      ▼
 3. Предпросмотр: извлечь текст с первой страницы
      │
      ▼
 4. Оценка качества текстового слоя (mode-detector)
      │  → score 0-1, список проблем, рекомендация
      │
      ▼
 5. Пользователь выбирает режим (текст / OCR / VLM)
      │  или принимает рекомендацию автоматически
      │
      ▼
 6. Пользователь выбирает диапазон страниц (from – to)
      │
      ▼
 7. Чтение файла через FileHandle → ArrayBuffer
      │
      ▼
 8. Извлечение контента по страницам:
      │
      ├─ Текстовый режим (text):
      │    PDF.js textContent → текст каждой страницы → чанки
      │
      ├─ OCR режим (ocr):
      │    Рендер → PNG → base64 → Vision LLM → Markdown с LaTeX
      │    → чанки Markdown
      │
      └─ VLM режим (vlm):
           Рендер → PNG → base64 → Vision LLM → идеи напрямую
      │
      ▼
 9. Проход 1: Извлечение идей (каждый чанк → LLM)
      │
      │  Prompt: извлечь идеи, вернуть JSON
      │  Результат всегда на русском языке
      │
      ▼
10. Дедупликация идей (по названию)
      │
      ▼
11. Проход 2: Построение связей (все идеи → LLM)
      │
      │  Типы: prerequisite, elaborates, contradicts, analogous, applies
      │
      ▼
12. Сохранение в IndexedDB + логирование
      │
      ▼
13. Отображение обновлённого списка идей / графа
```

## Спецификация LLM-промптов

> Подробные промпты будут в `docs/prompts.md` после фазы 0. Ниже — общая стратегия.

### Проход 1: Извлечение идей (текстовый режим)

```
Системный промпт:
  Ты — аналитик знаний. Извлекай из текста самостоятельные идеи,
  концепции, определения, утверждения, методы.
  Пиши результат на русском языке.

  Для каждой идеи укажи:
  - title: краткое название (до 10 слов)
  - summary: суть в 1-2 предложениях
  - type: definition | method | theorem | insight | example | analogy
  - depth: basic | medium | advanced
  - importance: 1-5 (насколько фундаментальна)
  - pages: номера страниц (из контекста)
  - quote: ключевая цитата (если есть)

  Правила:
  - НЕ дроби мелкие детали
  - НЕ дублируй повторяющиеся идеи
  - Если идея опирается на другую — укажи в requires[]

  Верни JSON массив идей.
```

### Проход 1: Извлечение идей (VLM-режим)

```
Системный промпт:
  Ты — аналитик знаний. На изображении — страница учебника/книги.
  Извлеки все идеи с этой страницы.
  Пиши результат на русском языке.
  Формат тот же, что и для текстового режима.
```

### Проход 2: Построение связей

```
Системный промпт:
  Дан список идей из книги. Определи связи между ними:

  Типы связей:
  - prerequisite: идея A нужна для понимания B
  - elaborates: идея A развивает/уточняет B
  - contradicts: идея A противоречит B
  - analogous: идея A аналогична B из другой области
  - applies: идея A применяется в B

  Верни JSON массив: [{source, target, type, description}]
```

## Решения по ключевым вопросам

### Хранение файлов: File System Access API

**Почему не IndexedDB для файлов:**
- Лимит IndexedDB: 50-500MB (зависит от браузера и диска)
- Типичный учебник: 10-100MB, DJVU: 5-50MB
- 10-20 книг могут заполнить лимит

**Почему не Native Messaging:**
- Требует установки отдельной программы на компьютер
- Разная установка под Windows/Linux/macOS
- Избыточно для нашей задачи

**Решение:** `FileSystemFileHandle` хранится в IndexedDB (~100 байт). При необходимости чтения — `handle.getFile()` → ArrayBuffer → рендер. Книга остаётся на диске,IndexedDB не перегружается.

** permission истекает** — обычно после закрытия браузера. При повторном открытии расширения — кнопка «Переподключить файл», которая запрашивает новый handle и обновляет запись в БД. Путь файла также хранится для справки (но через `resolve()` нельзя восстановить handle по пути).

### DJVU-рендер: стратегия

1. **Основной путь**: djvu.js (JavaScript-библиотека для рендера в canvas)
2. **Fallback**: Если djvu.js не справляется с конкретным файлом — предложить пользователю конвертацию через Linux-утилиты:
   - `djvups book.djvu book.ps && ps2pdf book.ps book.pdf`
   - `ddjvu -format=pdf book.djvu book.pdf`
3. **OCR fallback**: Если ни текстовый слой, ни VLM не дают результат — предложить `tesseract` через Native Messaging (далёкая перспектива)

### Интеграция с внешним ридером

Кнопка «Открыть в ридере» → копирует в буфер обмена команду:

```bash
zathura -f 42 /home/user/books/math_analysis.djvu
```

Пользователь вставляет в терминал одним кликом. Не требует установки native host, работает на любом Linux.

В перспективе: можно зарегистрировать `ideograph://` protocol handler через `registerProtocolHandler()`, но это требует desktop-файла и более сложной настройки.

### Мультиязычность книг

Язык книги не важен. В промпте указано: «пиши результат на русском языке». LLM (особенно GPT-4o, Claude) отлично извлекает идеи с английского, немецкого, французского и других языков и формулирует на русском. Не нужно делать перевод книги — только извлечение идей на русском.

### Автоопределение режима (text / OCR / VLM)

Алгоритм оценки качества текстового слоя (`mode-detector.ts`) анализирует извлечённый текст и возвращает числовую оценку (0-1) и рекомендацию. Упрощённая логика:

```
score = доля читаемых символов

Если garbage-символов > 10%   → score -= garbage * 0.5
Если обрывков слов > 30%      → score -= 0.3
Если пробелов > 85%          → score -= 0.3
Если математ. символов > 3   → score -= 0.15

Рекомендация:
  score < 0.3  → VLM (текстовый слой непригоден)
  score < 0.65 → OCR (текст частично повреждён, формулы)
  иначе         → TEXT (текст хорошего качества)

DJVU-штраф: если формат djvu и score < 0.8 → OCR
```

Пользователь всегда может изменить рекомендованный режим вручную.
