# Идеограф — Навигатор по идеям книг

Chrome-расширение для извлечения, структурирования и навигации по идеям из учебников и научных книг. Позволяет разбивать книгу на сеть связанных идей, отслеживать уровень освоения и быстро ориентироваться в материале.

## Проблема

Невозможно глубоко изучить все книги, которые хочется. Линейное чтение страница за страницей — медленно и неэффективно. Книги по одной теме (например, матанализу) на 80-90% пересекаются по содержанию. Ключевой навык — не прочитывать всё подряд, а **оперировать идеями**: понимать что уже знаешь, что новое, как идеи связаны между собой.

## Решение

Расширение, которое:
- Загружает PDF/DJVU книгу (через File System Access API, без хранения в IndexedDB)
- Анализирует выбранный диапазон страниц через LLM (текстовый или vision-режим)
- Извлекает идеи в структурированном виде (название, суть, тип, глубина, важность)
- Строит граф связей между идеей
- Позволяет отмечать статус каждой идеи (знакомо / не понятно / освоено)
- Даёт быстрый обзор книги за 10-15 секунд по предисловию или оглавлению

## Ключевые принципы

1. **Идеи, не страницы** — книга рассматривается как набор идей со связями, а не линейный текст
2. **VLM-экстракция** — книги без текстового слоя (сканы, DJVU) обрабатываются через vision-модели
3. **Нет встроенного ридера** — чтение во внешнем приложении (zathura и др.), расширение — «мозговой центр»
4. **Результат на русском** — извлечение идей из книги на любом языке, результат всегда на русском
5. **Селективный анализ** — обрабатываются только выбранные пользователем участки, не вся книга целиком

## Установка и настройка

### Установка расширения в Chrome

Идеограф — это Chrome-расширение Manifest V3. Расширение открывается по нажатию на иконку в тулбаре или через комбинацию `Alt+I`. Для установки потребуется собрать проект из исходного кода.

**Подготовка окружения:**

Убедитесь, что у вас установлены Node.js (версии 18+) и npm:

```bash
node --version   # должна быть >= 18
npm --version
```

**Сборка расширения:**

```bash
# Клонировать репозиторий
git clone https://github.com/NikasAl/ideograph.git
cd ideograph

# Установить зависимости
npm install

# Собрать расширение
npm run build

# Или запустить dev-сервер с hot-reload
npm run dev
```

После сборки в каталоге `dist/` появится собранное расширение.

**Установка в Chrome:**

1. Откройте Chrome и перейдите по адресу `chrome://extensions/`
2. В правом верхнем углу включите переключатель **«Режим разработчика»** (Developer mode)
3. Нажмите кнопку **«Загрузить распакованное расширение»** (Load unpacked)
4. Выберите каталог `dist/` проекта
5. Расширение появится в списке установленных расширений
6. Нажмите на иконку Идеографа в тулбаре (или `Alt+I`) — откроется интерфейс

> **Важно**: после каждого изменения исходного кода нужно пересобрать проект (`npm run build`) и нажать кнопку обновления на карточке расширения в `chrome://extensions/`. В dev-режиме (`npm run dev`) обновление происходит автоматически.

### Настройка API ключей

Идеограф использует LLM для извлечения идей из книг. Для работы необходим API-ключ хотя бы одного провайдера.

**Получение ключей:**

| Провайдер | Где получить ключ | Формат ключа |
|-----------|-------------------|-------------|
| **OpenRouter** | [openrouter.ai/keys](https://openrouter.ai/keys) | `sk-or-v1-...` |
| **z-ai** | Предоставляется при регистрации на платформе | Зависит от формата провайдера |

**Ввод ключа:**

1. Откройте Идеограф (нажмите на иконку или `Alt+I`)
2. Нажмите **Настройки** в правом верхнем углу
3. В разделе **«AI провайдеры»** введите API-ключ в соответствующее поле
4. В разделе **«Провайдер»** выберите активного провайдера
5. Нажмите **Сохранить**

> **Безопасность**: API-ключи хранятся локально в IndexedDB браузера и никогда не отправляются на сторонние серверы (кроме самого API провайдера при выполнении запросов).

### Выбор моделей

Идеограф использует три модели для разных этапов обработки. Все модели настраиваются в разделе настроек. Для удобства есть кнопки-пресеты, которые автоматически заполняют поля рекомендованными моделями.

| Параметр | Назначение | Рекомендация (OpenRouter) |
|----------|-----------|---------------------------|
| **Основная модель (activeModel)** | Извлечение идей из текста / Markdown. Должна хорошо понимать русский язык и работать с JSON. | `anthropic/claude-sonnet-4` |
| **OCR модель (ocrModel)** | Конвертация изображения страницы в Markdown с LaTeX-формулами. Vision-модель. | `google/gemini-2.0-flash-001` |
| **VLM модель (vlmModel)** | Полный визуальный анализ страницы — извлечение идей напрямую из изображения. | `anthropic/claude-sonnet-4` |
| **Fallback модели** | Резервные модели при ошибках основной (429, 502/503/529). Через запятую. | `google/gemma-4-26b-a4b-it:free,openai/gpt-4o-mini` |

**Почему три модели:** Разные этапы требуют разных возможностей. OCR-конвертация — простая задача для быстрой vision-модели (Gemini Flash). Извлечение идей — глубокое понимание контекста, лучше Claude. Визуальный анализ — мощная vision-модель. Одна модель для всего — либо слишком дорого, либо недостаточно качественно.

### Подключение книг

1. В Идеографе нажмите **«+ Добавить книгу»**
2. В появившемся диалоге выберите PDF или DJVU файл
3. Расширение автоматически:
   - Определит формат файла (по расширению `.pdf`, `.djvu`, `.djv`)
   - Посчитает количество страниц (PDF.js для PDF, DjVu.js для DJVU)
   - Извлечёт текст с нескольких страниц (до 5 равномерно распределённых) и оценит качество текстового слоя
   - Подберёт оптимальный режим экстракции (текстовый / OCR / VLM)
4. Книга появится в библиотеке с указанием формата, количества страниц и рекомендованного режима

> **Ограничение**: файлы книг **не копируются** в базу данных расширения. Идеограф хранит ссылку на файл (через File System Access API). Сам файл остаётся на диске. При перемещении/удалении файла нужно переподключить через кнопку на карточке книги.

### Модули

#### 1. Библиотека книг (`book-list.ts`)
- Список загруженных книг с metadata (название, автор, кол-во страниц, формат)
- Подключение книги через File System Access API (`showOpenFilePicker`)
- Автоопределение формата: PDF (`.pdf`) или DJVU (`.djvu`, `.djv`)
- Подсчёт страниц: PDF.js для PDF, DjVu.js для DJVU
- Мультистраничная оценка качества текстового слоя (до 5 страниц, равномерно распределённых)
- Персистентное хранение `FileSystemFileHandle` в IndexedDB (таблица `fileHandles`)
- Автоматическое восстановление handle из IndexedDB при перезагрузке расширения
- Кнопка «Переподключить» при потере permission
- Кнопка «Открыть в ридере» — запускает zathura на странице книги через NMH
- Удаление книги (из библиотеки, файл на диске не трогается)
- Бейджи на карточке: формат (PDF/DJVU), режим экстракции, наличие TOC

#### 2. Экстракция идей — 3 режима (`pipeline.ts`, `mode-detector.ts`)
- Выбор диапазона страниц для анализа
- Мультистраничная оценка качества текстового слоя (до 5 страниц)
- Подробный отчёт по качеству каждой страницы
- Три режима экстракции (подробнее см. [Режимы экстракции](#режимы-экстракции)):
  - **Текстовый (text)**: PDF.js/DjVu.js -> текст -> LLM
  - **OCR + Текстовый (ocr)**: страница -> изображение -> vision LLM -> Markdown с LaTeX -> text LLM -> идеи
  - **Полный визуальный (vlm)**: страница -> изображение -> vision LLM -> идеи напрямую
- Автоопределение режима по качеству текстового слоя (0-1)
- Пользователь может переключить режим вручную
- **Выбранный пользователем режим всегда учитывается** — пайплайн не переопределяет его
- Прогресс-бар с текущим этапом обработки
- Отмена анализа через `AbortSignal`
- Частичное сохранение при ошибке: извлечённые идеи + `lastAnalyzedPage` для возобновления
- Кеширование контекстных данных глав из TOC для injection в промпты

#### 3. Карточки идей (`idea-list.ts`)
- Список идей с фильтрами (по статусу, типу, знакомству, главе)
- Карточка каждой идеи: название, суть, тип, глубина, важность, страницы, цитата
- Редактирование заметок с автосохранением (debounced)
- Интерактивные статусы: Не изучал -> В процессе -> Освоено -> Применяю -> Не понятно
- Знакомство: Не знаком / Слышал / Знаю
- Пользовательские теги
- Кнопка «Контекст» — просмотр кэшированного текста страницы (OCR или текстовый слой)
- Кнопка «zathura» на каждой карточке — открывает книгу на нужной странице через NMH; при наличии цитаты показывает подсказку для поиска
- Удаление идей с подтверждением

#### 4. Граф идей (`idea-graph.ts`)
- Force-directed граф на vis-network (forceAtlas2Based layout)
- Цветовая кодировка по главам (15-цветная палитра)
- Форма узла по типу идеи: круг (definition), квадрат (method), ромб (theorem), треугольник (insight), звезды (example), шестиугольник (analogy)
- 5 стилей рёбер по типу связи (prerequisite — стрелка, analogous — пунктир, и т.д.)
- Размер узла = важность
- Drag-and-drop, zoom/pan
- Кнопка «Fit to view»
- Hover-тултипы с названием и типом идеи
- Клик по узлу -> карточка идеи (TODO)
- Фильтры и поиск по графу (TODO)

#### 5. Настройки (`settings-modal.ts`)
- API ключи: OpenRouter, z-ai
- Выбор активного провайдера
- Три отдельные модели: `activeModel` (text), `ocrModel` (vision для OCR), `vlmModel` (vision для VLM)
- Fallback модели: запятые-разделённый список
- Кнопки-пресеты для быстрого заполнения моделей (OpenRouter + z-ai)
- Задержка между запросами (`requestDelayMs`)
- Тема: светлая / тёмная / системная (с автоопределением)
- Детализация извлечения: низкая / средняя / высокая

#### 6. Оглавление книги — TOC (`toc-panel.ts`)
- **Извлечение из встроенных закладок (bookmarks):** без LLM, мгновенно. PDF — через `pdf.getOutline()`, DJVU — через `doc.getContents()`. Автоматическая конвертация в `TOCEntry[]` с учётом `pageOffset`.
- **Извлечение через LLM:** 3 режима (text / ocr / vlm), пакетная обработка (4 страницы/батч), для DJVU используются соответствующие DjVu.js рендереры.
- Иерархическое дерево (3 уровня) с сворачиваемыми группами (состояние сохраняется в localStorage)
- Автоматическое вычисление диапазонов страниц (`pageEnd`)
- Инлайн-редактирование: заголовок, номер страницы, уровень, добавление/удаление элементов
- Суммаризация глав (LLM, только уровень 1) — кнопка «Описать все главы»
- Калибровка смещения страниц (`pageOffset`): книга-страница N -> документ-страница M
- Кнопка «Анализировать главу» на каждом элементе TOC -> открывает AnalysisPanel с заполненным диапазоном
- Фильтрация идей по главе
- zathura-интеграция для каждого элемента TOC

#### 7. Бенчмарк моделей (`model-test.ts`)
- Загрузка списка моделей из OpenRouter (с ценами) или использование встроенного списка z-ai
- Фильтры: поиск по названию, максимальная цена, только vision, быстрые выборки (бесплатные, <$1, все)

- Поддержка пользовательских тестов (добавление/удаление, сохранение в localStorage)

- Экспорт результатов в JSON
- История тестов с очисткой

## Поддерживаемые форматы

### PDF (pdfjs-dist)

Полная поддержка PDF через PDF.js v4:

- **Текст:** извлечение текста через `getTextContent()` с каждой страницы
- **Рендеринг:** `render()` на OffscreenCanvas -> масштабирование -> base64 PNG
- **Закладки (outline):** `pdf.getOutline()` с рекурсивным обходом иерархии, конвертация document-page -> book-page через `pageOffset`
- **Количество страниц:** `pdf.numPages`

PDF.js работает через web worker (`pdf.worker.min.mjs` в `public/`). При каждом вызове создаётся копия ArrayBuffer, так как PDF.js отсоединяет (detaches) буфер.

### DJVU (djvujs-dist)

Полная поддержка DJVU через DjVu.js v0.5:

- **Текст:** извлечение через `page.getText()` (встроенный текстовый слой DJVU)
- **Рендеринг:** `page.getImageData()` -> OffscreenCanvas (масштабирование) -> base64 PNG
- **Закладки (NAVM):** `doc.getContents()` с парсингом URL формата `#N` (номер страницы) и рекурсивным обходом `children`
- **Количество страниц:** `doc.getPagesQuantity()`

DjVu.js подключается через vite-алиас (`djvujs-dist/library/src/index.js` -> `node_modules/djvujs-dist/library/src/index.js`), так как пакет не имеет поля `main`/`exports` в корневом `package.json`. Типовые объявления находятся в `src/types/djvujs-dist.d.ts`. Экстрактор (`djvu-extractor.ts`) используется динамически через `import()` для оптимизации размера чанка.

> **Оценка текстового слоя для DJVU:** при оценке через `mode-detector.ts` применяется штраф -0.15 к оценке, так как DJVU-текстовые слои часто содержат артефакты. Порог рекомендации текстового режима для DJVU повышен до 0.8 (вместо 0.65 для PDF).

## Режимы экстракции

Идеограф поддерживает три режима извлечения идей. Режим выбирается автоматически на основе оценки текстового слоя, но пользователь может переключить его вручную. Все три режима поддерживают как PDF, так и DJVU файлы.

### Текстовый режим (text)

**Когда использовать:** текстовый слой хорошего качества — текст читается, формулы корректны, OCR-ошибок нет.

**Как работает:**
1. PDF.js или DjVu.js извлекает текст с каждой страницы
2. Текст разбивается на чанки (~4000 символов, с учётом границ страниц)
3. Каждый чанк отправляется в текстовую LLM для извлечения идей
4. Извлечённые идеи дедуплицируются
5. Все идеи отправляются в LLM для построения связей (проход 2)

### OCR + Текстовый анализ (ocr)

**Когда использовать:** текстовый слой есть, но некачественный — формулы искажены, OCR-ошибки, частичный скан.

**Как работает:**
1. **Фаза 1 — OCR-конвертация:**
   - Каждая страница рендерится в изображение (OffscreenCanvas -> PNG) через соответствующий рендерер (PDF.js или DjVu.js)
   - Изображение отправляется в vision LLM (OCR модель) для конвертации в Markdown
   - Формулы восстанавливаются в LaTeX: `$$...$$` для блочных, `$...$` для inline
   - Результат кэшируется в `pageCache.ocrMarkdown`
2. **Фаза 2 — извлечение идей:**
   - Markdown разбивается на чанки (~5000 символов)
   - Каждый чанк -> текстовая LLM -> идеи
   - Дедупликация + построение связей

### Полный визуальный анализ (vlm)

**Когда использовать:** визуальный контент (чертежи, графики, диаграммы) или отсутствие текстового слоя.

**Как работает:**
1. Каждая страница рендерится в изображение через соответствующий рендерер
2. Изображение + промпт -> vision LLM (VLM модель) -> идеи
3. Дедупликация + построение связей

### Предпросмотр текстового слоя

При нажатии «Анализировать» Идеограф оценивает качество текстового слоя по нескольким страницам (до 5, равномерно распределённых по диапазону). Для каждой страницы вычисляется оценка 0-1 на основе: доля читаемых символов, наличие мусорных символов, доминирование пробелов, «сломанные» короткие слова, нераспознанные математические символы. Средневзвешенная оценка определяет рекомендацию режима.

## Пайплайн обработки

Общий поток для всех трёх режимов:

```
Выбор диапазона страниц
        |
        v
Оценка текстового слоя (multi-page, до 5 стр.)
        |
        v
Пользователь подтверждает / переключает режим
        |
        v
+-------+-------+-------+
|  text  |  ocr  |  vlm  |
+---+----+---+----+---+--+
    |        |        |
    v        v        v
PDF.js/  render->  render->
DjVu.js  VLM OCR   VLM extract
text     ->MD      ideas/page
  |        |
  v        v
LLM extract   LLM extract
ideas/chunk   ideas/chunk
  |              |
  +------+-------+
         |
         v
  Дедупликация идей (по title)
         |
         v
  Построение связей (pass 2, LLM)
         |
         v
  Привязка к главам (chapterId из TOC)
         |
         v
  Сохранение в IndexedDB
         |
         v
  Логирование в analysisLog
```

**Обработка ошибок:** при ошибке на любой странице:
- Сохраняются уже извлечённые идеи (partial save)
- Обновляется `lastAnalyzedPage` до последней успешно обработанной страницы
- Ошибка логируется в `analysisLog`
- Исключение пробрасывается наверх для отображения в UI

## Работа с оглавлением (TOC)

### Извлечение из встроенных закладок (bookmarks)

Для PDF и DJVU файлов, содержащих встроенные закладки, оглавление можно извлечь **без LLM** — мгновенно и бесплатно.

- **PDF:** `pdf.getOutline()` — рекурсивный обход дерева закладок, разрешение document-page в book-page через `pageOffset`
- **DJVU:** `doc.getContents()` — парсинг NAVM-чанка, URL формата `#N`, рекурсивный обход `children`

Результат конвертируется в `TOCEntry[]` с автоматическим вычислением `pageEnd` и построением иерархии через `parentId`. Кнопка «Из bookmarks» в TOC-панели.

### Извлечение через LLM

Для книг без встроенных закладок, либо если они неполные/неточные:

1. Пользователь указывает диапазон страниц оглавления
2. Выбирается режим извлечения (text / ocr / vlm)
3. Для DJVU файлов автоматически используются DjVu.js-рендереры (не PDF.js)
4. Страницы обрабатываются пакетами (4 страницы/батч)
5. LLM возвращает JSON с элементами: title, page, level, parentTitle
6. Пост-обработка: дедупликация, построение иерархии (по parentTitle + позиционный fallback), валидация

### Редактирование и калибровка

- **Инлайн-редактирование:** заголовок, номер страницы, уровень иерархии
- **Добавление/удаление** элементов
- **Калибровка pageOffset:** если номера страниц в оглавлении не совпадают с номерами страниц документа (часто в книгах с римской нумерацией предисловия), можно настроить смещение. Например, оглавление указывает страницу 15, а в PDF это страница 17 -> offset = 2.

### Привязка идей к главам

После извлечения идей пайплайн автоматически привязывает каждую идею к главе (уровень 1) через `chapterId`. Привязка идёт по совпадению `bookPage = ideaPage - pageOffset` в диапазоне `[chapter.page, chapter.pageEnd]`. Главные контекстные описания (summary) из TOC инжектируются в промпт извлечения идей для улучшения качества.

## AI-интеграция

### Провайдеры

| Провайдер | Класс | Endpoint | Особенности |
|-----------|-------|----------|-------------|
| **OpenRouter** | `OpenRouterProvider` | `openrouter.ai/api/v1` | Заголовки `HTTP-Referer`, `X-Title`; метод `listModels()` |
| **z-ai** | `ZAIProvider` | Пользовательский `baseUrl` | Заголовок `X-Z-AI-From: Z`; OpenAI-совместимый API |

API-интерфейс провайдера:
```typescript
interface AIProvider {
  chat(messages: ChatMessage[], options: ChatOptions): Promise<ChatResponse>;
  chatVision(messages: VisionMessage[], options: ChatOptions): Promise<ChatResponse>;
  listModels?(): Promise<ModelInfo[]>;
}
```

### Система fallback и retry

При вызове LLM используется двухуровневая система обработки ошибок:

1. **Транзиентный retry (429, 502, 503, 529):** до 2 повторных попыток на каждую модель с экспоненциальной задержкой (1с -> 2с -> 4с, max 10с)
2. **Перманентный fallback (400, 401, 403, 404, 422, «Provider returned error»):** немедленный переключение на следующую модель из `fallbackModels`

Все модели (основная + fallback) проходят через один и тот же цикл. Если все модели исчерпаны — выбрасывается исключение с описанием последней ошибки.

### Бенчмарк моделей

Встроенный инструмент для тестирования и сравнения моделей:

- Загрузка моделей из OpenRouter (с ценами) или использование списка z-ai
- 4 встроенных теста: простой ответ (10%), следование инструкциям (30%), JSON-формат (30%), извлечение идей (30%)
- Пользовательские тесты с сохранением в localStorage
- Ручная оценка (1-5 звёзд) для субъективных критериев
- Экспорт результатов в JSON для анализа

                                                       |

                                            +---------------------+

                                            |  --fork file.pdf    |
                                            +---------------------+

## Решения по ключевым вопросам

**Почему Chrome Extension, а не веб-приложение?** File System Access API и Native Messaging Host доступны только в контексте расширения. Это обеспечивает прямой доступ к файлам на диске и интеграцию с десктопными приложениями.

**Почему не хранить файлы в IndexedDB?** Книги (особенно DJVU с растровыми страницами) могут занимать сотни мегабайт. Хранение в IndexedDB привело бы к необоснованному расходу памяти и замедлению работы. Вместо этого используется File System Access API — расширение получает доступ к файлу по необходимости.

**Почему OffscreenCanvas, а не Canvas?** OffscreenCanvas работает в Web Worker и не требует DOM, что позволяет рендерить страницы без блокирования UI. Это критично для OCR и VLM режимов, где каждая страница рендерится перед отправкой в LLM.

**Почему Denormalized relations?** Связи (relations) хранятся прямо в объекте Idea, а не в отдельной таблице. Это устраняет JOIN-запросы при отображении карточек и графа, что важно для IndexedDB, не поддерживающего SQL JOIN. Цена — при удалении идеи нужно очистить обратные ссылки (запланировано).

**Почему DjVu.js, а не серверная конвертация?** DjVu.js позволяет работать с DJVU-файлами локально, без отправки на сервер. Это соответствует принципу приватности — файлы не покидают устройство. Кроме того, djvujs-dist работает полностью в браузере через Web Workers и OffscreenCanvas.
